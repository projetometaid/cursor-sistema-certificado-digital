# Relatório — 12/09/2025 - 07h41m

Este documento registra o que foi executado para deixar o fluxo de integrações 100% funcional (Safeweb, Assertiva, ViaCEP e consultas de CNPJ), além de orientar como este processo deve ser aplicado futuramente em um formulário público (site) consumindo o backend como BFF.

## Objetivo

- Garantir geração de protocolos e-CPF A1 e e-CNPJ A1 com experiência guiada no Admin.
- Centralizar integrações via backend para evitar CORS e padronizar validações/erros.
- Tornar o fluxo resiliente quando Assertiva estiver indisponível (fallback com dados do usuário).

## Alterações implementadas (resumo técnico)

- Backend
  - Segurança: `helmet`, `cors` restrito, `express-rate-limit` e uso de `role` somente do refresh payload.
  - Novas rotas de Integrações (`/integrations/*`) para atuar como BFF:
    - `POST /integrations/biometria` (Safeweb)
    - `POST /integrations/assertiva/cpf`
    - `POST /integrations/safeweb/consulta-previa-cpf`
    - `POST /integrations/safeweb/consulta-previa-cnpj`
    - `GET  /integrations/externos/cnpj/:cnpj` (ReceitaWS → fallback BrasilAPI)
    - `GET  /integrations/viacep/:cep`
  - Protocolos (`/protocols`): validações Zod, aceitação de CNH quando não há biometria, e fallback para `dataNascimento` no e-CNPJ (usa a informada; se ausente, tenta Assertiva; se Assertiva falhar, retorna erro claro de ausência de data).
  - Proxies locais (Safeweb/Assertiva) fixados em `127.0.0.1` e com logs ruidosos removidos.

- Frontend (Admin)
  - Tela “Novo Certificado” com dois fluxos dinâmicos e máscaras visuais (CPF, CNPJ, CEP, telefone, PIS, CEI, CAEPF, data, CNH).
  - e-CPF: após CPF → biometria (Safeweb), data (Assertiva), nome (consulta-prévia Safeweb), CEP (ViaCEP).
  - e-CNPJ: CPF do responsável → biometria (Safeweb), data (Assertiva ou manual), nome (consulta-prévia CPF). CNPJ → consulta-prévia CNPJ (Safeweb) confirma representante legal e define a razão social; endereço da empresa vem de `externos/cnpj` (não sobrescreve a razão social).
  - Razão social “fixada” apenas a partir da Safeweb; dados de endereço não alteram a razão social.

## Fluxos fim a fim

### e-CPF A1

1. Usuário informa CPF → backend chama Safeweb biometria e Assertiva (DOB).
2. Com CPF+DOB, consulta-prévia CPF (Safeweb) retorna nome oficial; exibe no formulário.
3. CEP preenche endereço via ViaCEP. Telefone/E-mail preenchidos.
4. CNH obrigatória somente se biometria negativa.
5. Envia `POST /protocols/ecpf` com dados; backend monta payload Safeweb e gera protocolo; cria PIX (Safe2Pay) referenciando o protocolo.

### e-CNPJ A1

1. CPF do responsável → biometria (Safeweb) e DOB (Assertiva ou manual). Consulta-prévia CPF preenche nome do responsável.
2. CNPJ (onBlur) → consulta-prévia CNPJ (Safeweb) confirma representante legal e define a razão social (mantida fixa).
3. Endereço da empresa é carregado via `externos/cnpj` e aplicado apenas aos campos de endereço.
4. Envia `POST /protocols/ecnpj` com `dataNascimento` do responsável (se enviada, não depende da Assertiva). Backend valida CPF/CNPJ na Safeweb e gera protocolo + PIX.

## Endpoints principais (BFF)

- Integrações
  - `POST /integrations/biometria` { cpf }
  - `POST /integrations/assertiva/cpf` { cpf }
  - `POST /integrations/safeweb/consulta-previa-cpf` { cpf, dataNascimento }
  - `POST /integrations/safeweb/consulta-previa-cnpj` { cnpj, cpfResponsavel, dataNascimento }
  - `GET  /integrations/externos/cnpj/:cnpj`
  - `GET  /integrations/viacep/:cep`
- Protocolos
  - `POST /protocols/ecpf`
  - `POST /protocols/ecnpj` (aceita `dataNascimento` do responsável)

## Formulário público (site) — como integrar

- Use o backend como BFF, chamando somente `/integrations/*` e `/protocols/*` para evitar CORS e expor credenciais.
- Sequência recomendada:
  - e-CPF: CPF → biometria; Assertiva (DOB) se disponível; consulta-prévia CPF (Safeweb) para nome; CEP via ViaCEP; CNH somente se biometria negativa; enviar para `/protocols/ecpf`.
  - e-CNPJ: CPF responsável → biometria; DOB (Assertiva ou manual); consulta-prévia CPF (Safeweb) para nome. CNPJ → consulta-prévia CNPJ (Safeweb) confirma representante e fixa razão social; endereço empresa via `externos/cnpj`; enviar para `/protocols/ecnpj` incluindo `dataNascimento`.
- Boas práticas:
  - Máscaras visuais e `onBlur` para disparo de integrações.
  - Rate limit no backend e validações com Zod.
  - Logs sem PII sensível; monitoração e alertas.
  - Tratamento de degradação: se Assertiva falhar, permita entrada manual da data (e-CNPJ), mantendo validações da Safeweb.

## Testes e troubleshooting

- Smoke: gerar e-CPF e e-CNPJ com CPF/CNPJ de teste, validando preenchimentos automáticos e PIX.
- Se a Assertiva retornar 500:
  - e-CNPJ: preencha manualmente a data; o backend aceita `dataNascimento` no corpo.
  - e-CPF: necessário DOB para consulta-prévia; se faltar, o backend retorna erro claro.
- Se razão social “piscar”:
  - Verifique que a UI está recebendo a razão da Safeweb e que os dados de endereço (externos) não a sobrescrevem (implantado).

### Usuários – Modelo, RBAC e Endpoints (09/09/2025 16:24)

Guia de referência do módulo de Usuários, autenticação, RBAC e como isso se conecta à Agenda.

## Visão Geral
- Cadastro e login com JWT (access e refresh tokens).
- RBAC simples: `admin`, `provider`, `viewer`.
- Rotas protegidas com middleware (`authenticate`, `requireRole`, `requireSelfOrAdmin`).
- Integração com Agenda: `schedulingEnabled` e ligação com `SchedulePolicy` por `userId`.

## Modelo de Usuário (mínimo e seguro)
- Essenciais:
  - `id` (UUID)
  - `fullName` (string)
  - `email` (normalizado em minúsculo, único)
  - `passwordHash` (bcrypt no MVP)
  - `role` (`admin` | `provider` | `viewer`)
  - `schedulingEnabled` (boolean)
  - `timezone` (ex.: `America/Sao_Paulo`)
  - `status` (`active` | `suspended`)
  - `createdAt`, `updatedAt`, `lastLoginAt`
- Opcionais:
  - `phone`, `tenantId`
  - `consent` { `ad_user_data`, `ad_personalization` }
  - `metadata` (objeto livre)

## RBAC e Guards
- `authenticate`: exige `Authorization: Bearer <accessToken>`; popula `req.user = { id, role }`.
- `requireRole('admin')`: só admin.
- `requireSelfOrAdmin(param)`: o usuário autenticado só acessa se `req.user.id === req.params[param]` ou for admin.

## Endpoints (HTTP)
- Autenticação (`src/interface/http/routes/auth.js`)
  - POST `/auth/register` – cria usuário.
    - DTO: `register.dto.js`
```json
{ "fullName":"Nome", "email":"a@b.com", "password":"********", "role":"provider", "timezone":"America/Sao_Paulo", "schedulingEnabled": false }
```
  - POST `/auth/login` – retorna `accessToken`, `refreshToken`.
    - DTO: `login.dto.js`
```json
{ "email":"a@b.com", "password":"********" }
```
  - POST `/auth/refresh` – troca refresh por novo access.
```json
{ "refreshToken":"<jwt>" }
```

- Usuários (`src/interface/http/routes/users.js`) [protegido]
  - GET `/users/:id` – `requireSelfOrAdmin('id')`
  - PUT `/users/:id` – atualiza perfil (nome, fuso, consent…) `requireSelfOrAdmin('id')`
    - DTO: `update-user.dto.js`
  - PATCH `/users/:id/role` – admin altera papel `requireRole('admin')`
    - DTO: `setRole`
  - PATCH `/users/:id/scheduling` – liga/desliga agenda do usuário `requireSelfOrAdmin('id')`
    - DTO: `setScheduling`

- Agenda (trecho) – políticas protegidas, disponibilidade pública
  - PUT `/schedules/:providerId/enabled` – `authenticate` + `requireSelfOrAdmin('providerId')`
  - PUT `/schedules/:providerId/policy` – `authenticate` + `requireSelfOrAdmin('providerId')`
  - GET `/schedules/:providerId/availability` – público (para widget)

## Exemplo rápido (curl)
- Registro e login:
```bash
curl -sX POST http://localhost:3000/auth/register \
  -H 'Content-Type: application/json' \
  -d '{"fullName":"Ana","email":"ana@example.com","password":"12345678","role":"provider"}'

curl -sX POST http://localhost:3000/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"email":"ana@example.com","password":"12345678"}'
```
- Perfil (autenticado):
```bash
ACCESS="<accessToken>"; USER_ID="<id>"
curl -sX GET http://localhost:3000/users/$USER_ID -H "Authorization: Bearer $ACCESS"
```
- Scheduling on/off (autenticado):
```bash
curl -sX PATCH http://localhost:3000/users/$USER_ID/scheduling \
  -H 'Content-Type: application/json' -H "Authorization: Bearer $ACCESS" \
  -d '{"enabled": true}'
```

## Tokens e Segurança
- Access: TTL padrão `15m`. Refresh: `7d`. Configurável via env.
- Nunca logar senhas/tokens. Força mínima de senha: 8+.
- Refresh com rotação futura (opcional); por ora validação simples via `JwtAuthTokenService`.

## Configuração (ENV)
- `JWT_SECRET`, `ACCESS_TTL`, `REFRESH_TTL`, `BCRYPT_ROUNDS`
- Ver `src/infrastructure/security/JwtAuthTokenService.js` e `BcryptPasswordHasher.js`.

## Integração com Agenda
- `schedulingEnabled` controla se o usuário atende.
- `SchedulePolicy` fica em coleção própria, ligada por `providerId` (== `userId`).
- Controllers de usuários/agenda fazem a ponte (evitar acoplamento no core).
